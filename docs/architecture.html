MoneyPrinterTurbo

Architecture, Features & Cost Guide

An end-to-end map of what this repository does, how a video is produced stage by stage, and exactly where money is (and isn't) spent when you run it.

What it is

MoneyPrinterTurbo turns a single topic or keyword into a finished, HD short-form video — the kind you'd post to TikTok, YouTube Shorts, or Instagram Reels — with no manual editing. From one subject line it writes the script, picks stock-footage search terms, synthesizes a narration voice, generates timed subtitles, downloads matching b-roll, and composites everything (clips + narration + subtitles + background music) into one or more rendered .mp4 files.

It is a layered Python application. The service layer is the true engine, and it is exposed two ways over the exact same code:

FastAPI service

main.py → app/asgi.py. REST endpoints for creating tasks, polling status, uploading media, and streaming/downloading results. Good for automation and scripting.

Streamlit WebUI

webui/Main.py. A point-and-click interface that imports the service layer directly — it is not merely an HTTP client of the API.

Built on FastAPI, Streamlit, MoviePy 2.x + FFmpeg, edge-tts, faster-whisper, OpenAI-compatible SDKs, and optional Redis. Runtime configuration lives in config.toml (auto-created from config.example.toml), not environment variables.

The high-level goal

Automate the entire short-video content pipeline so that producing dozens of faceless, narrated short videos costs minutes of compute instead of hours of human editing — and, with the optional cross-post integration, can publish them automatically.

Concretely it aims to:

  • Remove every manual step between "I have an idea" and "I have a publishable MP4": scripting, voiceover, subtitling, footage sourcing, and editing.
  • Stay provider-agnostic and cheap. ~20 LLM providers, multiple TTS engines, and two stock-media sources are pluggable via config — including paths that cost literally nothing.
  • Scale horizontally. A task queue with configurable concurrency (in-memory or Redis-backed) lets one instance crank out many videos in parallel.
  • Be operable by non-developers through the WebUI, while remaining fully scriptable through the API.

System architecture

Two entry points feed one service layer. A task manager schedules work onto threads; the orchestrator runs the pipeline and calls out to pluggable provider services. State and queueing each have in-memory and Redis-backed variants, toggled together by enable_redis.

The generation pipeline

task.start(task_id, params, stop_at) runs eight ordered stages. The stop_at argument ("script" / "terms" / "audio" / "subtitle" / "materials" / "video") lets the lighter /scripts, /terms, /audio, and /subtitle endpoints reuse the same function for partial runs. Each stage reports progress into the state store.

Full feature list — what it can do

Content generation (LLM)

Script writing

From subject, language, paragraph count, custom user prompt, and optional custom system prompt. Strips <think> reasoning blocks.

Search-term generation

Derives stock-footage keywords from the subject + script for media retrieval.

Social metadata

Title, caption, and hashtags for TikTok, YouTube Shorts, Instagram Reels, Facebook Reels.

~20 LLM providers

OpenAI, AIHubMix, Moonshot, Azure, Qwen, DeepSeek, Gemini, Grok, Ollama, g4f, OneAPI, Cloudflare, MiniMax, MiMo, ERNIE, ModelScope, Pollinations, LiteLLM (100+ via gateway).

Audio & voice

Edge TTS (default)

Free Microsoft Edge voices, shown as "Azure TTS V1". Also yields subtitle timing data. free

Azure Speech "V2"

Higher-quality neural voices via Azure Speech SDK (needs [azure] keys). paid

SiliconFlow / Gemini / MiMo TTS

Additional cloud TTS engines (CosyVoice2, Gemini TTS voices, Xiaomi MiMo TTS).

Custom audio

Upload your own narration track instead of TTS.

No-voice mode

Silent placeholder audio to drive the timeline for music-only videos.

Voice rate control

Speed up / slow down narration.

Subtitles

Edge timing (fast)

Reuses timestamps emitted by Edge TTS — no transcription needed. free

Whisper (accurate)

Local faster-whisper transcription + correction against the script. Falls back automatically if edge produces nothing.

Full styling

Font, size, color, stroke, position (top/center/bottom/custom %), background, rounded background. Rendered with Pillow — no ImageMagick needed.

Media & footage

Pexels / Pixabay

Stock video search + download by term, with API-key rotation and TLS verification. free API

Local materials

Upload your own videos/images via API or WebUI.

Image → clip

Turns still images into short zooming video clips.

Material caching

Downloaded clips cached under storage/cache_videos for reuse across tasks.

Video composition

Aspect ratios

Portrait 9:16, landscape 16:9, square 1:1 (API).

Concat modes

Sequential or random clip ordering.

Transitions

Fade in/out, slide in/out, shuffle.

Clip fitting

Trim, resize, loop clips to match audio duration.

Background music

None / random / custom BGM mixed under narration; uploadable BGM library.

Multiple variants

Generate N video variants per task in one run.

Hardware encoders

libx264 (default), NVENC, AMF, QSV, MediaFoundation, VideoToolbox — auto-fallback to libx264.

Serving, tasks & ops

Stream & download

Byte-range streaming and direct download of outputs, path-traversal guarded.

Task lifecycle

Create, list (paginated), query status/progress, delete tasks + files.

Concurrency & queue

Configurable max_concurrent_tasks + max_queued_tasks (429 when full).

Redis option

Swap in-memory state/queue for Redis-backed for multi-process scale.

Auto cross-post

Optional Upload-Post integration to TikTok & Instagram after render. paid

Docker / GPU

docker-compose runs API + WebUI; GPU variants for Whisper/encoding.

Known gap: the WebUI lists TikTok, Bilibili, and Xiaohongshu as video source labels, but the backend downloader only implements Pexels and Pixabay. Any unrecognized online source value falls back to the Pexels search path.

End-to-end deep dive: how one video gets made

This traces the full video generation feature — the flagship path — from an API/WebUI request to a downloadable MP4. Source of truth: app/services/task.py:start().

0

Request & scheduling

POST /videos validates a VideoParams object (subject, language, voice, aspect, counts, subtitle styling, source…). A task_id is created, initial state stored, and the job handed to the Task Manager, which runs it immediately if under max_concurrent_tasks or queues it (HTTP 429 if the queue is full). The client receives the task_id and starts polling GET /tasks/{id}. Progress is set to 5%.

1

Script — generate_script()

If params.video_script is empty, llm.generate_script() calls the configured provider (config.app["llm_provider"]) with the role prompt + subject + language + paragraph count. The response is cleaned (<think> blocks stripped, newlines normalized). A non-empty custom script skips the LLM entirely. ← LLM cost incurred here (unless local/free provider). Progress → 10%. stop_at="script" returns now.

2

Search terms — generate_terms()

Only for online sources (skipped when video_source="local"). llm.generate_terms() produces ~5 English keywords for stock search. Script + terms + params are persisted to storage/tasks/<id>/script.json. ← second (small) LLM call. stop_at="terms" returns now.

3

Audio — generate_audio()

Three branches: (a) a valid custom_audio_file is used as-is (no TTS, no subtitle timing); (b) otherwise voice.tts() synthesizes audio.mp3 via the selected engine and returns a sub_maker carrying word timing; (c) no-voice mode produces silent audio. Audio duration is measured (get_audio_duration). ← TTS cost here for paid engines; Edge TTS is free. Progress → 30%. stop_at="audio" returns now.

4

Subtitles — generate_subtitle()

Skipped if disabled or no sub_maker. With provider edge, timing from the TTS step is written straight to subtitle.srt (fast, free). With whisper (or as an automatic fallback), local faster-whisper transcribes the audio and subtitle.correct() aligns it to the known script. The result is validated. ← runs locally, no API cost (uses CPU/GPU). stop_at="subtitle" returns now.

5

Materials — get_video_materials()

For local source, uploaded videos/images are preprocessed (images become zooming clips). Otherwise material.download_videos() searches Pexels/Pixabay per term, downloading enough footage to cover audio_duration × video_count, caching it. API keys rotate across a pool to dodge rate limits. ← stock APIs are free tier; bandwidth/storage are the only cost. Progress → 40–50%. stop_at="materials" returns now.

6

Combine clips — video.combine_videos()

Per variant (video_count): clips are trimmed to max_clip_duration, resized to the target aspect, ordered (sequential/random), looped to fill the narration length, and joined with the chosen transition into combined-N.mp4. ← local CPU/GPU via FFmpeg. Progress climbs toward 75%.

7

Final render — video.generate_video()

The combined track is muxed with narration, optional BGM (mixed under the voice), and styled subtitles (rendered via Pillow), encoded with the selected codec (HW encoder or libx264 fallback) into final-N.mp4. ← the most CPU/GPU-intensive step; local only. Progress → ~100%.

8

Optional cross-post + completion

If Upload-Post is configured with auto-upload, each final video is pushed to TikTok/Instagram. The state store is marked COMPLETE with all output paths (videos, combined, audio, subtitle, materials, cross-post results). The client's next poll returns URLs servable via /tasks, /stream, or /download. ← Upload-Post is a paid third-party service if enabled.

Where the costs are

The project is designed so the core pipeline can run essentially free. Costs appear only at specific external touch-points, and almost every one has a free alternative. Here is every place money can be spent:

Cost centerWhen it appliesCost profile
LLM (script + terms + metadata) Every full run makes ~2 calls (script + ~5-word terms); social metadata is a 3rd if used. paid for hosted APIs · free with Ollama/g4f/Pollinations/free tiers. Tiny per video — see below.
TTS (narration) Every run unless using custom audio or no-voice. free with Edge TTS (default) · paid for Azure Speech / SiliconFlow / Gemini / MiMo TTS (per character).
Stock media (Pexels / Pixabay) Any online video source. free API (rate-limited; key rotation supported). Only egress/storage cost.
Compute / rendering Every run — combine + encode is CPU/GPU heavy. self-hosted Your machine or a rented VM/GPU. The dominant real cost at scale.

Redis Only if enable_redis=true. optional Free self-hosted; small cost for managed Redis.
Upload-Post cross-post Only if cross-posting enabled. paid third-party SaaS subscription.
Bandwidth & storage Always — clips downloaded, outputs stored under storage/. infra Grows with footage cache + number/length of videos.

LLM cost in detail

The LLM workload per video is small: a script of a few short paragraphs plus a one-line list of ~5 keywords. That's roughly a few hundred input tokens and a few hundred output tokens per video — call it ~1–2k total tokens including the system prompt.

Rough order of magnitude (verify current prices with the provider): on a cheap hosted model in the "mini/flash" tier, ~1–2k tokens is a fraction of a US cent per video — typically well under $0.01. On a premium frontier model it might reach a few cents. Either way, for the text generation step, LLM cost is rarely the bottleneck. The default config points at low-cost "mini/flash"-tier models precisely for this reason.

To drive LLM cost to $0: use ollama (local models), pollinations (free public tier), provider free tiers (e.g. Gemini flash), or g4f (reverse-engineered endpoints — disabled by default, use at your own risk).

The real cost at scale is compute, not APIs. Video encoding (combine + final render, especially with multiple variants or Whisper on CPU) is what consumes minutes of CPU/GPU per video. If you batch hundreds of videos, your VM/GPU bill dwarfs the per-video LLM and TTS costs.

Running it for (almost) $0

The default-friendly free configuration:

[app]
video_source     = "pexels"        # free API key
llm_provider     = "ollama"        # or "pollinations" / a free tier
subtitle_provider = "edge"         # reuse TTS timing — no Whisper, no API
enable_redis     = false           # in-memory queue/state

# voice_name in the request → an Edge TTS voice (free, "Azure TTS V1")
# no [azure] / siliconflow keys needed
# upload_post_enabled = false
With this stack the only cost is the electricity/VM time to render video. No LLM bill, no TTS bill, no subtitle bill, no publishing bill — just a free Pexels/Pixabay API key and your own compute.

Trade-offs: Ollama needs local model weights + RAM/VRAM; Edge TTS voices are good but not as natural as premium neural voices; edge-timing subtitles are slightly less precise than Whisper. For most faceless short-video use cases this is more than enough.
